interface UpdateAction {

**Step 2 — Fix row-matching in applyUpdateSurgical (~line 423):**

After the existing `voucherMatch` check, add a group-type match guard for EANs:

```typescript
if (campaignMatch && voucherMatch) {
  // For EANs tab: if group_type is provided, also check that the correct
  // group-type column is non-empty on this row.
  if (action.tab === EANS_SHEET && action.group_type) {
    const row = rows[r] as unknown[];
    const inclusionsVal = String(row[2] ?? "").trim();  // col 2: Inclusions Group
    const exclusionVal  = String(row[3] ?? "").trim();  // col 3: Exclusion Group
    const isInclusionRow = inclusionsVal !== "";
    const isExclusionRow = exclusionVal !== "";

    if (action.group_type === "inclusion" && !isInclusionRow) continue;
    if (action.group_type === "exclusion" && !isExclusionRow) continue;
  }
  targetRowIdx = r;
  break;
}
```

Apply the **same fix** to the fallback xlsx path in `applyUpdate` (~line 831) — it has identical row-matching logic. Add the same group-type guard there.

**Step 3 — Wire no-match offer in message handler (~line 1013):**

Change the update failure branch to detect EANs no-match and produce an offer-to-insert:

```typescript
if (action?.operation === "update") {
  const result = await applyUpdate(action as UpdateAction);
  if (result.success) {
    displayText += "\n\nBrief is updated: [Download the latest Brief.xlsx here](.data/Brief.xlsx)";
  } else if (
    result.error === "Record not found." &&
    (action as UpdateAction).tab === EANS_SHEET

    // Offer to insert instead of generic failure
    const ua = action as UpdateAction;
    const groupLabel = ua.group_type === "inclusion" ? "inclusions" : ua.group_type === "exclusion" ? "exclusion" : "matching";
    displayText += `\n\nNo existing ${groupLabel} row found for "${ua.voucher_name ?? "that voucher"}". Would you like to add it? Reply "yes" to insert, or provide the corrected details.`;
  } else {
    displayText += "\n\n⚠️ The brief could not be saved automatically. Please contact your admin.";
  }
}
```

Add a new section "## PERFORMING UPDATES (EANs Voucher Dashboard)" to instructions.txt, placed after the new "## PERFORMING ADDS (EANs Voucher Dashboard)" section added in plan 01-01.

The section must cover:

**Row identification:**
The EANs Voucher Dashboard has one row per voucher-group combination. A voucher may have both an inclusions row and an exclusion row. The `group_type` field disambiguates which row to target:
- "inclusions" / "inclusion group" → target the row where Inclusions Group is non-empty → emit `"group_type": "inclusion"`
- "exclusion" / "exclusion group" → target the row where Exclusion Group is non-empty → emit `"group_type": "exclusion"`

**Updatable fields:** Voucher Name, Inclusions Group, Exclusion Group (any of the three cols 1-3 can be changed)

**No-match behavior:**
If the system responds with a no-match offer ("No existing [inclusions/exclusion] row found..."), the user must explicitly reply "yes" or "confirm" before the bot inserts. Do not insert automatically from a failed update. Wait for confirmation, then emit an add action block using the original data.

**Action block format** (emit after validation passes):
```
<action>
{
  "operation": "update",
  "tab": "Included-Excluded EANs- Voucher",
  "voucher_name": "<voucher name to match — case-insensitive substring>",
  "group_type": "<inclusion | exclusion>",
  "field": "<exact column header: 'Voucher Name', 'Inclusions Group', or 'Exclusion Group'>",
  "value": "<new value>"
}
</action>
```

**IMPORTANT notes:**
- Always include `group_type` for EANs update actions — omitting it will cause the wrong row to be updated when a voucher has both row types
- Use the exact column header text as it appears in the sheet (verify from [BRIEF DATA] header line for the EANs tab)
- campaign_name is not used for EANs updates — omit it from the action block
- The system confirms success with "Brief is updated" and appends the download link automatically

Also update the "## RESPONSE FORMAT" section to add a subsection "After a Successful EANs Update:" showing the same Before/After format but using Voucher + Group Type + Field labels (no Campaign line).

Follow the case-insensitive / partial matching pattern for Voucher Name (same as Promo-Voucher). If the match is ambiguous (multiple rows for same voucher name without clear group type), ask the user to specify "inclusions" or "exclusion" group.
